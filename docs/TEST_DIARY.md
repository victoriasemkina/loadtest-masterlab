## 2. Spike Test — Первая проблема

**Дата:** 2026-03-01  
**Тест:** `SpikeSimulation`  
**Длительность:** 6 минут  
**Нагрузка:** 10 → 100 → 10 users/sec (скачок в 10 раз)

### Результаты (первый запуск)

| Метрика | Значение | Порог | Статус |
|---------|----------|-------|--------|
| Всего запросов | 20,726 | — | ✅ |
| Ошибки | **2,530 (12%)** | ≤ 5% | ❌ **FAIL** |
| Response Time 95% | 778ms | ≤ 2000ms | ✅ |
| BUILD | **FAILURE** | — | ❌ |

### Типы ошибок

| Ошибка | Количество | % |
|--------|-----------|---|
| `Connection refused: getsockopt` | 2,530 | 88% |
| `No attribute named 'transferId'` | 334 | 12% |

### Анализ проблем

#### 🔴 Проблема 1: Connection refused (88%)

**Причина:** Spring Boot приложение **упало** на пике нагрузки.

**Диагноз:** Недостаточно памяти (JVM heap).

**Доказательство:**
- Ошибки начались на 300-й секунде (пик нагрузки 100 users/sec)
- Приложение перестало отвечать на порт 8080 
- Gatling получил Connection refused

#### 🔴 Проблема 2: No attribute 'transferId' (12%)

**Причина:** `transferId` не сохранился в сессию Gatling.

**Диагноз:** Следствие падения приложения — `executeTransfer()` не выполнился.

**Доказательство:**
- Ошибки начались ПОСЛЕ Connection refused 
- transferId не был сохранён → checkTransferStatus() получил пустое значение

### Вывод

> ❌ **Тест провален.** Система **не выдержала** резкий скачок нагрузки.

**Но это ХОРОШО!** Мы нашли проблему **до продакшена**.

## 3. Анализ и исправление

### Исправление 1: Увеличить память JVM

**Файл:** `app/pom.xml`

```xml
<plugin>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-maven-plugin</artifactId>
    <configuration>
        <jvmArguments>-Xmx512m -Xms256m</jvmArguments>
        <mainClass>io.github.victoriasemkina.loadtest.LoadTestApplication</mainClass>
    </configuration>
</plugin>
```
> Обоснование: Выделяем до 512MB памяти для стабильности под нагрузкой.

### Исправление 2: Проверить сохранение transferId

**Файл:** `src/main/java/simulations/scenarios/transfer/TransferScenarios.java`

```java
public static ChainBuilder executeTransfer() {
    return exec(
        http("POST /api/transfer/execute")
            .post("/api/transfer/execute")
            .body(StringBody("{\"fromAccount\": \"#{fromAccount}\", \"toAccount\": \"#{toAccount}\", \"amount\": #{amount}}"))
            .asJson()
            .check(status().is(201))
            .check(jsonPath("$.transferId").saveAs("transferId"))  // ← Сохраняем
    ).pause(Duration.ofMillis(100));
}

public static ChainBuilder checkTransferStatus() {
    return exec(
        http("GET /api/transfer/status")
            .get("/api/transfer/status/#{transferId}")  // ← Используем
            .check(status().is(200))
    ).pause(Duration.ofMillis(100));
}
```
**План проверки**
1. Перезапустить приложение с новыми настройками памяти 
2. Повторить Spike тест 
3. Ожидать: 0% ошибок, BUILD SUCCESS

### 5. Ключевые выводы
🎯 Главный урок 
> Упавший нагрузочный тест — это НЕ неудача. Это УСПЕХ.

Без Spike теста:
- Проблема обнаружилась бы при виральном трафике 
- Приложение упало бы в продакшене 
- Пользователи не смогли бы совершить перевод

Со Spike тестом:
- Проблема найдена на тестовом окружении 
- Исправлена до релиза (увеличили память)
- Система готова к реальным скачкам трафика
